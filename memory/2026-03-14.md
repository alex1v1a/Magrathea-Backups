

## Recovery Tool: OpenClaw API Key Troubleshooting Checklist

**Created**: 2026-03-14 21:28 CDT
**Purpose**: Standard diagnostic procedure for API key authentication failures

### Symptoms
- 401/403 errors from all model providers
- Config shows correct keys but errors show old/different keys
- "Incorrect API key provided" errors
- Gateway starts but AI requests fail

### Diagnostic Steps (In Order)

#### Step 1: Verify Config File
```bash
cat ~/.openclaw/openclaw.json | jq '.models.default.apiKey'
```
- Check if key matches expected value
- Note: Config may be correct but overridden by other caches

#### Step 2: Check Hidden Environment Cache
```bash
cat ~/.openclaw/.env
```
- **CRITICAL**: This file overrides config file settings
- If present, takes precedence over openclaw.json
- Common cause of "config changes not taking effect"

#### Step 3: Check Agent Auth Profiles
```bash
cat ~/.openclaw/agents/main/agent/auth-profiles.json
```
- Third cache layer for agent-specific credentials
- Must be valid JSON (check for missing quotes)
- Can be malformed if edited manually

#### Step 4: Validate JSON Format
```bash
python3 -m json.tool ~/.openclaw/agents/main/agent/auth-profiles.json > /dev/null && echo "Valid JSON" || echo "Invalid JSON"
```
- auth-profiles.json must be valid JSON
- Common issue: Missing quotes around keys/values

#### Step 5: Test API Keys Directly
```bash
# Moonshot
curl -s -H "Authorization: Bearer $MOONSHOT_API_KEY" https://api.moonshot.ai/v1/models

# OpenAI
curl -s -H "Authorization: Bearer $OPENAI_API_KEY" https://api.openai.com/v1/models

# Anthropic
curl -s -H "x-api-key: $ANTHROPIC_API_KEY" https://api.anthropic.com/v1/models
```
- Tests if keys are valid (not expired/revoked)
- Bypasses OpenClaw entirely
- If these fail, keys are invalid - not a caching issue

#### Step 6: Restart Gateway
```bash
pkill -f 'openclaw-gateway'
nohup openclaw gateway --port 18789 > /tmp/openclaw-gateway.log 2>&1 &
sleep 5
openclaw gateway status | grep RPC
```
- Must restart after ANY credential file changes
- Verify RPC probe shows "ok"

#### Step 7: Check Logs for Errors
```bash
tail -50 /tmp/openclaw-gateway.log | grep -E '401|auth|error'
```
- Look for specific error messages
- Note which provider is failing
- Check if old key suffix appears in errors

### Common Issues & Solutions

| Issue | Cause | Solution |
|-------|-------|----------|
| Config changes not applying | `.env` file overriding | Update or remove `.env` file |
| JSON parse errors | Malformed auth-profiles.json | Use Python/json.tool to validate/fix |
| Keys valid but 401 errors | Invalid/expired keys | Regenerate from provider dashboards |
| RPC probe failed | Gateway not starting | Check logs for config errors |
| Old key in errors | Multiple cache layers | Update ALL three locations |

### Key Insight
OpenClaw 2026.3.x has a regression (GitHub #34656) where API keys are cached in multiple locations that persist across restarts. The `.env` file is the primary hidden cache that overrides the main config.

### Files to Never Ignore
1. `~/.openclaw/.env` - Environment overrides (HIGHEST PRIORITY)
2. `~/.openclaw/agents/main/agent/auth-profiles.json` - Agent cache
3. `~/.openclaw/openclaw.json` - Main config (LOWEST PRIORITY)

### Verification Checklist
- [ ] Config file has correct keys
- [ ] `.env` file updated or removed
- [ ] `auth-profiles.json` is valid JSON with correct keys
- [ ] Gateway restarted after changes
- [ ] RPC probe shows "ok"
- [ ] Direct API curl tests pass
- [ ] No 401 errors in logs

### When All Else Fails
If all cache locations are correct but keys still fail:
1. Keys are invalid/expired (regenerate from providers)
2. OpenClaw bug requires developer intervention
3. Complete reinstall may be necessary
